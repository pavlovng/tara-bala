# Техническое задание (Production — Full Detailed)
## Веб-приложение: Калькулятор Тара Балы (True Lahiri / Chitrapaksha)

---

# 1. Введение

## 1.1. Назначение

Данный документ описывает требования к разработке веб-приложения,
которое рассчитывает показатель **Тара Бала** из ведической астрологии (джйотиш).

Документ написан так, чтобы быть понятным разработчику без знаний астрологии.

---

## 1.2. Ключевая идея

Приложение:

1. Определяет положение Луны:
   - на момент рождения пользователя
   - на текущий момент

2. Делит круг 360° на 27 равных сегментов (накшатры)

3. Определяет:
   - в каком сегменте находится Луна при рождении (личная накшатра)
   - в каком сегменте Луна сейчас (текущая накшатра)

4. Вычисляет расстояние между ними → это и есть Тара Бала

---

# 2. Термины (упрощённо)

## 2.1. Эклиптическая долгота
Угол (0–360°), определяющий положение объекта на небе.

## 2.2. Луна
Нас интересует только её долгота.

## 2.3. Сидерический зодиак
Система координат, где используется корректировка — **аянамша**.

## 2.4. Аянамша (Ayanamsha)
Число (в градусах), которое вычитается из тропической долготы.

В проекте используется:
**True Lahiri / Chitrapaksha**

---

# 3. Основная математика

## 3.1. Формула

siderealLongitude = tropicalLongitude - ayanamsha

---

## 3.2. Деление на накшатры

- Всего: 27
- Размер: 360 / 27 = 13.333333°

```js
const NAKSHATRA_SIZE = 360 / 27;
```

---

## 3.3. Определение накшатры

```js
function normalize360(x) {
  return ((x % 360) + 360) % 360;
}

function getNakshatraIndex(longitude) {
  return Math.floor(normalize360(longitude) / NAKSHATRA_SIZE);
}
```

---

## 3.4. Расчёт Тара Балы

```js
function getTaraNumber(janma, current) {
  return ((current - janma + 27) % 27) + 1;
}

function getTaraClass(tara) {
  return ((tara - 1) % 9) + 1;
}
```

# 8. Часовые пояса

## 8.1. Подход: фиксированное смещение

Вместо IANA-имён часовых поясов (Europe/Moscow и т.д.) используются **фиксированные UTC-смещения** (offset:+3, offset:+4, offset:+5:30 и т.д.).

**Причина:** IANA-часовые пояса содержат исторические данные о переходах на летнее время (DST). Для даты рождения в 1988 году IANA может вернуть UTC+5 вместо ожидаемых UTC+4, что приведёт к ошибке в расчёте сидерической долготы Луны. Фиксированное смещение гарантирует, что 08:30 в UTC+4 всегда конвертируется в 04:30 UTC, независимо от исторического DST.

## 8.2. Список поддерживаемых смещений

UTC+0 … UTC+12 (с шагом 1), а также полуторачасовые:
UTC+3:30 (Иран), UTC+4:30 (Афганистан), UTC+5:30 (Индия), UTC+5:45 (Непал), UTC+6:30 (Мьянма), UTC+9:30 (Австралия ЦТ), UTC-1 … UTC-12.

## 8.3. Конвертация local → UTC

```
UTC = local_time - offset
```

Для получения UTC используется `Date.UTC()` с вычетом смещения в миллисекундах. Никаких вызовов `toLocaleString` — чистая арифметика.

# 10. Проверка границ накшатр

## 10.1. Порог

Порог чувствительности: **0.033°** (~2 минуты дуги).

Если позиция Луны в накшатре ближе чем 0.033° к началу или концу сегмента — считается, что Луна на границе.

## 10.2. Определение точного времени смены

Бинарный поиск с точностью до секунды:
1. Коarse: поиск с шагом 1 минута в окне ±15 минут от текущего JD
2. Fine: 20 итераций бинарного поиска

## 10.3. Отображение предупреждения

Формат: «Луна при рождении находилась на границе накшатр **X** и **Y**. Смена произошла в **HH:MM:SS**.»

Время смены отображается в **том же часовом поясе, что и данные рождения**.

# 12. Edge cases

- **Граница накшатры** — см. раздел 10
- **Нет времени рождения** — ошибка с просьбой указать максимально приближённое время
- **Неправильный timezone** — не применимо (используются фиксированные смещения)
- **Неизвестные координаты** — валидация: широта [-90, 90], долгота [-180, 180]

---

# 13. Производительность

- Кеш текущей накшатры **не используется** — пересчитываем каждый раз
- WASM + эфемеридные данные (~12 МБ) загружаются с CDN при старте
- После загрузки WASM — мгновенный расчёт
